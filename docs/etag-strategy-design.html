Combined ETag Strategy - Design Document

Goal: Minimize GraphQL API calls by using two lightweight ETag "guards" before deciding whether to run expensive batch queries.

The Problem with Single ETag Approach

Scenario: A CI check fails on PR #123

Event Updates PR.updated_at? PR List ETag Detects?
Developer commits code to PR ✓ Yes ✓ Yes
CI check starts running ✗ No ✗ No ← PROBLEM!
CI check FAILS ✗ No ✗ No ← PROBLEM!
Developer reviews the PR ✓ Yes ✓ Yes
Issue: PR List ETag misses CI status changes. We need a second guard.

Guard 1: PR List ETag Check

What It Does

Checks if PR metadata has changed by comparing ETag of the PR list endpoint.

Endpoint Used

GET /repos/{owner}/{repo}/pulls?state=open&sort=updated&direction=desc

How It Works

  1. First call: Fetch PR list, store ETag from response header
  2. Subsequent calls: Send If-None-Match header with stored ETag
  3. Response:
    • 304 Not Modified → No PR metadata changed, cost: 0 points
    • 200 OK → Something changed, cost: 1 REST point

What Changes Does Guard 1 Detect?

Event Updates ETag? Why?
New commits pushed to PR branch ✓ Yes Updates head_sha and updated_at
PR title edited ✓ Yes Updates title field
PR body/description edited ✓ Yes Updates body field
Labels added/removed ✓ Yes Updates labels array
Review submitted (approved/rejected) ✓ Yes Updates reviewDecision
PR opened/closed/reopened ✓ Yes Updates state and updated_at
Assignee changed ✓ Yes Updates assignees array
CI check starts/completes/fails ✗ No Commit check data stored separately ← Guard 1 misses this!

Implementation Example

// Cache for repo-level PR list ETags
const repoListETags = new Map<string, string>();
// Key: "owner/repo", Value: ETag string

// Example initial call
const response1 = await gh("api -i /repos/owner/repo/pulls?state=open");
// Response: HTTP/2.0 200 OK
// Etag: W/"abc123"
repoListETags.set("owner/repo", 'W/"abc123"');

// Example subsequent call
const response2 = await gh("api -i -H 'If-None-Match: W/\\"abc123\\"' /repos/owner/repo/pulls?state=open");
// Response: HTTP/2.0 304 Not Modified
// No body returned, cost: 0 points ✅

// Example with change
const response3 = await gh("api -i -H 'If-None-Match: W/\\"abc123\\"' /repos/owner/repo/pulls?state=open");
// Response: HTTP/2.0 200 OK
// Returns updated PR list, new ETag: W/"xyz789"
// Cost: 1 REST point
↓
Result from Guard 1: We know which repos have PR metadata changes

Guard 2: Commit Status Check

What It Does

Checks if CI status has changed for individual PRs by checking each PR's head commit.

Endpoint Used

GET /repos/{owner}/{repo}/commits/{head_sha}/status

Why We Need Guard 2

Guard 1 (PR List ETag) does NOT detect CI status changes because:

  • CI checks are stored on commit.statusCheckRollup, not on PR object
  • PR.updated_at is only updated when PR metadata changes, not when commit statuses update
  • Therefore, PR List ETag returns 304 even when CI fails!

How It Works

  1. For each PR with active CI: Use the PR's head_sha to query commit status
  2. Store ETag: Keep an ETag for each commit SHA
  3. Conditional request: Send If-None-Match with stored ETag
  4. Response:
    • 304 Not Modified → CI status unchanged, cost: 0 points
    • 200 OK → CI status changed, cost: 1 REST point

What Changes Does Guard 2 Detect?

Event Updates Commit ETag?
CI check starts running ✓ Yes
CI check PASSES ✓ Yes
CI check FAILS ✓ Yes ← This is what Guard 2 catches!
External status check updates (e.g., security scan) ✓ Yes
Deploy preview URL becomes available ✓ Yes

Important: Only Check PRs with Active CI

We don't need to check commit status for PRs where CI is already "passing" or "failing" and no new commits have been pushed.

Optimization: Only check commit status for PRs where ciStatus === "pending" or a new commit was recently pushed.

Implementation Example

// Cache for commit-level ETags
const commitETags = new Map<string, string>();
// Key: "owner/repo/sha", Value: ETag string

// Example PR with pending CI
const pr = { owner: "owner", repo: "repo", number: 123, headSha: "a1b2c3d", ciStatus: "pending" };

// Check commit status with ETag
const commitKey = `${pr.owner}/${pr.repo}/${pr.headSha}`;
const storedETag = commitETags.get(commitKey);

const response = await gh(
`api -i -H 'If-None-Match: ${storedETag ?? ''}' /repos/${pr.owner}/${pr.repo}/commits/${pr.headSha}/status`
);

if (response.includes("304 Not Modified")) {
  // CI status unchanged, cost: 0 points ✅

}
// Parse new CI status
const ciData = JSON.parse(response.split('\n\n')[1]);
const newETag = extractETag(response);
commitETags.set(commitKey, newETag);

Key Takeaway

Both guards together ensure complete coverage:

  • Guard 1 catches all PR metadata changes (commits, edits, reviews)
  • Guard 2 catches CI status changes that Guard 1 misses
  • Combined Only GraphQL batch when something actually changes
  • Result: 90%+ reduction in GraphQL points for typical workloads
